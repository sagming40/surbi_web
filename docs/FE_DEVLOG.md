### 겪었던 이슈들

**1. Hot Reload와 Hot Restart를 혼동함**
`main()` 안에 임시로 넣어둔 테스트 코드(`print`문)가 Hot Reload를 해도 콘솔에 전혀 안 찍혔다. 처음엔 코드가 잘못된 줄 알았는데, 원인은 **`main()`이 앱을 처음 켤 때 딱 한 번만 실행되는 함수**라는 것이었다. Hot Reload는 "이미 켜져 있는 앱의 화면을 다시 그리는" 동작이라 `main()`을 다시 타지 않는다. 완전 재시작(정지 → F5)을 해야 `main()`이 다시 실행된다는 걸 배웠다.

### 오늘 배운 것 / 느낀 점
- Hot Reload는 "이미 켜진 앱의 위젯을 갈아끼우는 것"이고, Hot Restart는 "앱을 껐다 켜는 것"이다. 앱 시작 시점에만 도는 코드(`main`, `initState` 등)를 건드렸다면 Reload로는 절대 확인할 수 없다.

## 2026-07-0? · 워크플로우 문서 Git 추적 중단 (.gitignore) 🔍 날짜 확인 필요

### 오늘 한 일
- 팀원 공유가 불필요하다고 판단해 워크플로우 문서를 `.gitignore`로 추적 중단
- `main`에서 `git rm --cached` 처리 후 push

### 겪었던 이슈들

**1. `git merge main` 시 rename/delete 충돌 발생**
`main`에서 문서 추적을 중단하고 push한 뒤, 작업 중이던 feature 브랜치에서 `git merge main`을 했더니 충돌이 났다. 원인은 **양쪽이 같은 파일을 서로 다르게 건드리고 있었기 때문**이다. `main`은 "이 파일 삭제(추적 중단)" 상태인데, feature 브랜치는 그 사이 파일명을 `v3.2 → v3.3`으로 **rename**한 상태였다. Git 입장에선 "삭제된 파일의 새 이름"이라는 게 성립하지 않으니 어느 쪽을 따라야 할지 판단을 못 한 것.
최종 목표(더 이상 추적하지 않는 것)에 맞춰 feature 브랜치에서도 `git rm --cached [파일명]`을 실행해 `main` 쪽 결정을 따르게 하고, `git commit`으로 병합을 마무리한 뒤 push했다.

**2. PowerShell `echo`로 `.gitignore`에 추가했더니 규칙이 먹통**
`echo "패턴" >> .gitignore`로 패턴을 추가했는데, 파일에 줄은 분명히 들어갔는데도 Git이 그 규칙을 전혀 인식하지 못했다. 원인은 **PowerShell이 파일을 UTF-16으로 저장**하면서 텍스트가 깨진 것. 눈으로 보기엔 파일이 있고 내용도 있는 것 같지만, Git이 읽을 수 있는 인코딩이 아니었다. 깨진 줄을 지우고 VS Code에서 직접 타이핑해 저장(UTF-8)하니 정상 작동했다.

### 오늘 배운 것 / 느낀 점
- **여러 브랜치가 같은 파일을 동시에 다르게 건드리면(이름 변경 vs 삭제) 병합 시 반드시 충돌이 난다.** `.gitignore` 같은 저장소 전역 설정 변경은, 가능하면 다른 브랜치들이 그 파일을 안 건드리는 시점에 처리하는 게 안전하다.
- PowerShell에서 파일에 텍스트를 추가할 땐 인코딩 이슈를 항상 의심할 것. 특히 한글·특수문자가 들어가면 그냥 에디터에서 직접 타이핑하는 게 확실하다.

### 오늘 한 일

- Step 3 건물 탐색 지도 모드 구현 완료 → **PR #5 오픈**

### 겪었던 이슈들

**1. 카카오맵 403 Forbidden — "제품 사용 설정" 토글을 놓침**
JavaScript 키 발급 + Web 플랫폼 도메인 등록까지 전부 마쳤는데도 콘솔엔 `kakao is not defined`, Network 탭엔 `403 Forbidden`이 떴다. 도메인 재확인, 하드 리프레시 등 할 수 있는 건 다 해봤지만 그대로였다.
원인은 카카오 디벨로퍼스의 **"제품 설정 > 카카오맵" 메뉴에 별도의 사용 ON/OFF 토글**이 있다는 것. 이게 꺼져 있었다. 키 발급·도메인 등록과는 **완전히 별개의 체크리스트 항목**이라 놓치기 딱 좋았다. 카카오 API는 "키 발급 + 도메인 등록 + **제품별 사용 설정 활성화**" 총 3단계를 다 확인해야 한다.

**2. `HtmlElementView` 리사이즈 시 지도 타일이 깨짐**
브라우저 창 크기를 바꾸거나 BottomSheet를 띄워서 지도 컨테이너(div) 크기가 변하면, 카카오맵이 새 크기를 인식하지 못하고 회색 여백 + 워터마크만 반복 표시됐다.
원인은 **카카오맵 JS SDK가 최초 렌더링 시 컨테이너 크기를 딱 한 번만 측정**하고, 이후 변화를 자동으로 따라가지 않는다는 것. `ResizeObserver`로 컨테이너 크기 변화를 감지해서 `kakao.maps.Map.relayout()`을 자동 호출하도록 구현해 해결했다.

**3. 마커 자동 포커싱(`setBounds`) 했더니 지도가 대한민국 전체로 튕김**
건물 마커 3개가 다 보이도록 `LatLngBounds` + `setBounds()`로 자동 줌/이동을 구현했는데, 첫 화면 로드 시 지도가 망원동이 아니라 **대한민국 전체 범위로 확 줌아웃**됐다. `lat`/`lng` 순서 같은 좌표값 자체는 정상이었다.
원인은 타이밍이었다. `HtmlElementView`(플랫폼 뷰)의 실제 브라우저 DOM 크기가 **완전히 확정되기 전에 `setBounds()`가 호출**되어, 카카오맵이 크기 계산에 실패하고 기본값(전체 지도)으로 fallback한 것으로 추정된다. 마커 생성 로직 진입 시 `Future.delayed(300ms)` + `map.relayout()`을 먼저 호출해서 컨테이너가 자리 잡을 시간을 확보한 뒤 `setBounds()`를 부르니 정상 동작했다.

### 미해결 / 보류

**BottomSheet 드래그 시 배경 지도가 같이 흔들림 — 롤백 처리**
마커를 탭하면 뜨는 BottomSheet를 아래로 드래그해 닫을 때, 배경 지도가 같이 위아래로 흔들리며 따라 움직였다. 총 4가지를 시도했다:
1. `pageTransitionsTheme` 변경
2. `RepaintBoundary`로 지도 격리
3. `Stack` + `AnimatedPositioned`로 직접 오버레이 구현
4. `Column` + `Expanded`로 물리적 분리 배치

전부 실패했고, 일부는 오히려 터치가 씹히거나 콘텐츠가 아예 안 뜨는 **더 심각한 문제로 악화**됐다.
추정 원인은 `HtmlElementView`(Platform View)가 Flutter의 일반 렌더링 파이프라인과 **별도로 브라우저 DOM 합성 레이어에서 그려진다**는 점이다. 이 두 렌더링 방식이 애니메이션 중에 충돌하는 것으로 보이며, Flutter Web + Platform View 조합에서 보고되는 알려진 렌더링 순서 한계로 추정된다.
**최종 결정**: 기능 손실이 없는 `showModalBottomSheet` 기본 방식으로 롤백. 팀 공유용 이슈 문서를 노션에 별도 게시하고 팀 논의로 넘겼다. (React였다면 달랐을지 팀장님께 확인 필요 — React vs Flutter Web 초기 논의와 연결됨)

### 오늘 배운 것 / 느낀 점
- **외부 SDK 연동은 "코드가 맞나"보다 "설정이 다 켜졌나"를 먼저 의심해야 한다.** 403은 코드 문제가 아니라 권한 문제라는 신호였는데, 코드만 들여다보느라 시간을 많이 썼다.
- 웹에서 지도 같은 **Platform View**를 쓴다는 건, Flutter가 그리는 세계와 브라우저가 그리는 세계 **두 개를 겹쳐놓는 것**과 같다. 두 세계는 서로의 존재를 잘 모른다. 크기 변화도, 애니메이션도 자동으로 동기화되지 않는다.
- **무리한 해결보다 실용적 선택이 나을 때가 있다.** 4가지 시도가 전부 상황을 악화시켰을 때, 계속 파는 대신 롤백하고 팀 논의로 넘긴 판단은 옳았다고 생각한다.

> 🔗 [PR #6 — Task 3-4 ~ 3-7 [Step 4] UI 일괄](https://github.com/sagming40/surbi_web/pull/6)

### 오늘 한 일
- Step 4-1 AI 종합 점수 게이지 + SHAP 시각화 UI 구현 완료

### 겪었던 이슈들

**1. SHAP 차트에서 숫자와 막대 위치가 어긋남**
숫자 값을 `Align(alignment: isPositive ? centerRight : centerLeft)`로 막대 방향에 맞춰 정렬했는데, 막대 끝과 숫자 위치가 계속 어긋나 보였다.
원인은 **기준점이 서로 달랐던 것**이다. 막대 자체는 비율(ratio)만큼만 그려지는데, `Align`의 기준은 **전체 폭**이다. 그러니 막대가 30%만 그려져 있어도 숫자는 100% 지점(맨 끝)에 붙는 셈.
해결은 발상을 바꾸는 쪽으로 했다. SHAP 원조 스타일인 "막대 끝에 숫자 붙이기"를 포기하고, `Row(mainAxisAlignment: MainAxisAlignment.spaceBetween)`으로 **요인명·수치를 같은 줄에 나란히 배치**하는 "라벨처럼 보여주기" 방식을 채택했다.

**2. `Center` + `SingleChildScrollView` 조합에서 상단 여백이 들쭉날쭉**
`body`를 `Center(child: SingleChildScrollView(child: Column(...)))` 구조로 짰더니, 콘텐츠가 화면보다 작을 때(전체화면)는 `Center`가 위쪽에 큰 여백을 만들고, 콘텐츠가 화면보다 클 때(창을 줄여 스크롤이 생길 때)는 여백이 사라졌다. 창 크기에 따라 레이아웃이 불안정해진 것.
원인은 **개념 자체가 안 맞았던 것**이다. `Center`는 "남는 공간을 가운데로 밀어 넣는" 위젯인데, 스크롤 가능한 콘텐츠는 애초에 "남는 공간"이라는 개념이 성립하지 않는다. `Center`를 제거하고 `Padding` → `SingleChildScrollView` → `Column` 순서로 단순화했다.

**3. 카드 사이 `SizedBox`를 빠뜨려 섹션이 겹쳐 보임**
"예상 성과" 카드와 "점수 상세 분석" 카드 사이에 넣기로 했던 `SizedBox(height: 20)`을 실수로 누락해서, 두 카드가 거의 붙어 보였다. 처음엔 위의 `Center` 문제 때문인 줄 알았는데, `Center`를 제거한 뒤에도 간격 문제가 남아 있어서 재확인 후에야 원인을 찾았다.

### 오늘 배운 것 / 느낀 점
- **"증상 하나 = 원인 하나"로 성급히 단정하지 말 것.** 이번엔 서로 다른 두 원인(레이아웃 정렬 + 여백 누락)이 겹쳐서 하나의 증상으로 보였다. 첫 원인을 고쳤는데 증상이 "덜해졌을 뿐 남아 있다면", 그건 원인이 하나 더 있다는 신호다.
- 위젯이 의도대로 안 나올 때, **그 위젯이 원래 무슨 일을 하는 물건인지**부터 다시 생각하면 답이 나오는 경우가 많다. `Center`는 애초에 스크롤과 어울리는 물건이 아니었다.

실제로는 `area_provider`/`building_provider`는 각 Step 브랜치에서 이미 개별 정리가 끝난 상태였고, `score_provider`만 누락돼 있었다. 결국 브랜치 분리 자체가 불필요했고(Step 4 소관 파일이므로), 현재 브랜치(`feature/frontend-step4-ui`)에서 그대로 처리했다.

**2. Task 3-4에서 B안 원칙 위반이 또 재발해 있었음**
`score_provider.dart`를 점검하다가, `step4_score_page.dart`가 여전히 `StatelessWidget`으로 되어 있고 종합 점수(75)·예상 월매출·SHAP 값이 **화면에 직접 하드코딩**된 채로 Task 3-4가 커밋·PR까지 진행됐다는 걸 발견했다.
추정 원인: `score_provider.dart`가 이미 컴파일 에러 상태였어서, 위젯(게이지·SHAP 차트)을 만들 때 "일단 값 넣어서 그리기부터" 하다가 **이후 Provider 연결로 마무리하는 단계를 누락**한 것으로 보인다. Step 1 검색창 자동완성 때와 **완전히 같은 성격의 실수 패턴**이다.
`ConsumerWidget` 전환 + `scoreResultProvider` 연결로 해결하고, **"PR 제출 전 View 파일에 하드코딩 값 없는지 최종 검색"**을 체크리스트 항목으로 추가했다.

**3. `Border(left:)`만 지정했는데 나머지 방향에 검은 테두리가 생김**
섹션 카드에 왼쪽 컬러 바만 주려고 `Border(left: BorderSide(...))`만 지정했더니, `top`/`right`/`bottom`이 **지정 안 한 기본값(검은 1px 테두리)으로 자동 설정**됐다. 특히 창을 줄였을 때 모서리에서 얇은 검은 선이 두드러졌다.
`Border()` 생성자는 지정하지 않은 방향을 "없음"이 아니라 **기본 테두리로 채운다.** `borderRadius`와 단방향 `Border` 조합 자체가 시각적으로 불안정한 조합이었다. `border` 대신 `Step4ScorePage`와 동일한 `boxShadow` 방식으로 전환하고, 색 구분은 카드 배경색 + 아이콘 원형 배지로 대체했다. (디자인 일관성도 같이 확보)

**4. `const`를 붙였다가 `invalid_constant` 컴파일 에러**
섹션 헬퍼 함수의 `TextStyle`에 `const`를 붙였더니 에러가 났다. 원인은 `color: accentColor`가 **함수 호출마다 값이 달라지는 파라미터(런타임에 결정)**인데, `const`는 **컴파일 타임에 값이 100% 고정된 것**에만 붙일 수 있기 때문. `const`를 제거해 해결했다. Task 3-4에서 `accentColor` 관련으로 같은 원리를 배웠는데 **또 반복한 케이스**다.

**5. `withOpacity()` deprecated 경고 재발**
Task 3-4에서 겪었던 정밀도 손실 이슈로 인한 `withOpacity()` deprecated 경고가, 새로 작성한 `report_viewer.dart` 헤더 텍스트 색상 코드에서 그대로 재발했다. `withOpacity(0.7)` → `withValues(alpha: 0.7)`로 전체 교체.

**6. `child:` 라벨을 빠뜨렸더니 레이아웃이 통째로 붕괴**
하단 고정 버튼을 가운데 정렬하려고 `Center(...)`를 추가하다 `child:` 라벨을 빠뜨려서 `extra_positional_arguments_could_be_named` 에러가 났다. 이 상태로 Hot Reload/Restart를 시도했더니 **AppBar와 본문 카드가 전부 사라지고 버튼만 화면 정중앙에 떠 있는** 비정상 렌더링이 나타났다.
추정 원인: 컴파일 에러가 난 상태에서 재시작을 시도하면, Flutter가 **마지막으로 성공했던 상태를 어중간하게 유지한 채** 화면을 그리는 경우가 있는 것으로 보인다. 해당 수정을 롤백하고 `width: double.infinity`로 유지했다. (버튼 가운데 정렬은 `Container`의 `alignment` 속성으로 추후 재시도 예정)

### 검토했으나 적용하지 않은 것

**LLM 보고서 텍스트 길이 대응 (Hero 애니메이션 카드 확대)**
"LLM 응답 길이가 가변적"이라는 우려로 카드 확대(Hero 애니메이션) 방식을 검토했는데, 확인해보니 현재 `SingleChildScrollView` + `Column` 구조 자체가 **이미 텍스트 길이와 무관하게 자동으로 늘어나고 스크롤 처리되는 구조**였다. 별도 UI 추가 없이 충분하다고 판단해 보류. 실제 텍스트 길이는 Task 4-2 API 명세 협의 시 재확인 예정.

**보고서에 체크리스트 섹션 포함 여부**
워크플로우 문서상 "체크리스트 섹션"이 보고서 화면에 포함된 것처럼 적혀 있었으나, 실제로는 `ChecklistItem` 모델과 Task 3-7이 **완전히 분리된 별도 화면**으로 설계돼 있었음을 재확인. 보고서는 3섹션만 구성하기로 확정.

### 오늘 배운 것 / 느낀 점
- 오늘 가장 큰 교훈은 **"코드가 이렇게 생겼다"고 보여주는 참고 자료(문서, 이전 대화 내용 등)도 항상 실물(에디터·터미널·git 히스토리)로 재검증해야 한다**는 것이다. 문서만 믿고 브랜치까지 팔 뻔했다.

  - 진행률처럼 원본에서 파생되는 계산값 → 별도 파생 Provider(`checklistProgressProvider`)로 분리

### 겪었던 이슈들

**1. 같은 이름의 Provider가 두 파일에 존재 (잠재적 폭탄)**
착수 전 확인해보니 `score_provider.dart`에 **예전 방식(단순 `Provider`)의 `checklistProvider`**가 이미 남아 있는 상태에서, 신규 `checklist_provider.dart`에도 **같은 이름의 `StateNotifierProvider`**를 만들어버렸다. 당장은 두 파일을 동시에 import하지 않아서 에러가 안 났지만, 나중에 화면에서 둘 다 참조하면 **ambiguous import 에러**가 날 잠재 위험이었다.
원인은 예전 버전이 상태 변경이 안 되는 단순 `Provider`라 체크 토글 구현 자체가 불가능한 구조였는데, 그걸 인지하지 못하고 새 Provider를 별도 파일에 만들어버린 것. `score_provider.dart`에서 구버전 `checklistProvider` 블록과 관련 import를 삭제하고, 신규 `StateNotifierProvider` 버전만 **단일 소스**로 유지했다. (refactor 커밋으로 선행 처리)

**2. 블록만 지우고 import는 남겨서 `unused_import` 경고**
위 이슈를 해결하면서 `checklistProvider` 블록(사용하는 부분)만 지우고 파일 상단의 `import '.../checklist_item.dart';`는 그대로 둔 채 커밋했다. `flutter analyze`에서 `unused_import` warning이 떠서 기존 6개 이슈가 **오히려 7개로 늘었다.** import 문도 함께 삭제해서 해결.

**3. `SizedBox`의 `height`를 늘렸는데 버튼 두께가 안 변함**
보고서 화면 하단 버튼(지원사업/체크리스트)의 두께를 키우려고 `SizedBox`의 `height`를 52 → 60으로 늘렸는데 아무 변화가 없었다.
원인은 **`Row`의 cross axis 특성**이었다. `Row`는 기본적으로 cross axis(세로) 방향으로 자식 위젯이 **원하는 크기만큼만** 차지하도록 둔다. 그래서 `OutlinedButton`이 부모(`SizedBox`)의 높이를 무시하고 자기 콘텐츠 크기만큼만 렌더링된 것. `Row`에 `crossAxisAlignment: CrossAxisAlignment.stretch`를 추가해 자식이 부모 높이만큼 늘어나도록 수정했다. `height` 숫자는 **늘어나는 폭을 조절하는 보조 역할일 뿐이고, `stretch`가 실질적인 해결책**이었다. 최종적으로 `height: 50` + `stretch`로 확정.

**4. Firestore 미연동 스텁 안내를 SnackBar로 하면 스팸이 됨**

### 겪었던 이슈들

**1. `StatefulShellBranch`의 기본 위치엔 파라미터화된 경로를 못 씀**
처음엔 `/step4/:buildingId/report`처럼 각 branch 경로에 `buildingId`를
그대로 포함시켰는데, 앱 실행 즉시 assertion 에러가 났다:
`"The default location of a StatefulShellBranch cannot be a parameterized
route"`. go_router는 branch를 처음 열 때의 "기본 위치"를 알아야 하는데,
그 경로에 채워야 할 파라미터가 있으면 값을 알 방법이 없어서 막는 것.
해결: `buildingId`는 **부모 GoRoute에서 한 번만** 받고, 세 branch는
파라미터 없는 상대경로(`report`/`policies`/`checklist`)만 갖도록 구조를
한 겹 더 감쌌다 — `/step4/:buildingId` 밑에 `routes: [StatefulShellRoute...]`.

**2. 부모의 `redirect`가 자식 간 탭 전환마다 매번 재실행됨**
Phase 2 완료 직후엔 문제없어 보였는데, Phase 3에서 탭 버튼을 눌러보니
"정부 지원사업"·"체크리스트"를 눌러도 화면이 전혀 안 바뀌었다(콘솔 에러도
없고 `InkWell` 리플은 정상적으로 뜸). 원인은 최초 진입용으로 걸어둔
`redirect: (context, state) => '${state.matchedLocation}/report'`가,
report/policies/checklist **전부의 조상 라우트**에 붙어 있었기 때문.
자식 중 어디로 이동하든 이 부모 라우트가 매칭 스택에 포함되고, go_router는
그때마다 `redirect`를 **다시 평가**한다. 그 결과 탭을 아무리 눌러도 매번
`/report`로 강제 복귀되고 있었다.
`state.matchedLocation`(항상 부모 경로 `/step4/$id`로 고정) 대신
`state.uri.path`(브라우저에 실제 표시되는 전체 경로)로 비교해서, **정확히
부모 경로 그 자체일 때만** report로 보내고 이미 자식 경로에 있으면
`null`을 반환해 아무 것도 안 하도록 조건을 추가해 해결.

**3. 증상은 같아 보였지만 원인이 세 겹으로 다른 문제였음**
탭이 안 눌리는 증상을 조사하던 중, 별개로 Step2 화면에 게이지 배경처럼
보이는 회색 원이 겹쳐 뜨는 현상도 같이 발생해서 원인을 한 덩어리로
오인했다. 실제로는:
- ① 회색 원 → 그냥 `InkWell` 리플 이펙트였음 (버그 아님)
- ② "로딩이 안 끝나고 멈춤" → 구조 변경 후 Hot Reload만 하고 완전
  재시작을 안 해서 생긴 잔상 (완전 재시작으로 해결)
- ③ "탭 눌러도 안 바뀜" → 위 2번 이슈(redirect 재실행)가 진짜 원인

**증상 하나 = 원인 하나로 성급히 묶지 말 것**(Task 3-4에서도 학습했던
내용의 재확인). 겹쳐 보이는 문제는 하나씩 분리해서 검증해야 한다.

**4. "화면은 바뀌는데 주소창만 안 바뀌는" 것처럼 보이는 현상 — 실제로는 지연**
디버깅 중 `location.href`를 직접 콘솔에 찍어 확인하는 습관을 들였는데,
탭 전환(단순 `goBranch` 1홉)에서는 실제 위치와 화면 표시가 항상 정확히
일치했다. 반면 Step3→Step4 **최초 진입**(push+redirect+branch진입이
겹치는 복잡한 한 번)에서만 화면 표시가 한 박자 늦게 따라오는 것처럼
보인 적이 있었다. 처음엔 크롬의 "과도한 history API 호출 제한
(Throttling navigation to prevent the browser from hanging,
crbug.com/882238)"을 의심했으나, 콘솔에 해당 경고 문구 자체가 없었고
이후 탭 전환은 전부 정확했던 것으로 보아 무관한 것으로 결론. **의심 가는
현상은 검색어부터 정확히 잡아야 한다** — `pushState`로 검색해서 안 나온
걸 "관련 로그 없음"으로 오판할 뻔했다가, 실제 크롬 경고 문구
("Throttling navigation...")로 다시 검색해서야 이 크롬 매커니즘의
정확한 트리거 조건을 알게 됐고, 결국 이번 케이스와는 무관하다고 판단했다.

### 오늘 배운 것 / 느낀 점
- go_router의 `redirect`는 "이 라우트에 처음 들어올 때 한 번"이 아니라,
  **이 라우트가 매칭 스택에 포함될 때마다** 재평가된다. 조상 라우트에
  redirect를 걸 땐 "자식 어디로 이동해도 이 redirect를 다시 타게 된다"는
  전제하에, 반드시 "지금 정말 리다이렉트가 필요한 상황인지"를 조건으로
  걸러야 한다.
- `state.matchedLocation`과 `state.uri.path`는 다르다. 전자는 "이 라우트가
  선언된 경로 패턴", 후자는 "지금 브라우저에 실제로 있는 전체 경로". 조상
  라우트의 redirect 조건문에선 후자를 써야 자식 위치를 구분할 수 있다.
- 화면 표시 이상 증상을 겪을 때 `location.href`를 콘솔에 직접 찍어보는 게
  "화면이 진짜 잘못됐는지, 화면 그리기만 늦은 건지"를 가장 빠르게 구분하는
  방법이었다.
- 브라우저/엔진 관련 증상을 검색으로 검증할 땐, 내가 임의로 지은 키워드가
  아니라 **콘솔에 실제로 뜨는 문구 그대로**를 검색해야 한다. 문구를 잘못
  짚으면 "검색해봤는데 없더라"는 결론 자체가 틀린 근거가 된다.
